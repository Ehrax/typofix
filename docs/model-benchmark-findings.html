Typofix Model Benchmark Findings

Captured from the live benchmark run on 2026-07-07. The benchmark is an opt-in XCTest and uses the local Typofix config for provider keys.

Current recommendation: keep llama-3.3-70b-versatile as the conservative fast-fix default for real usage, even though llama-3.1-8b-instant won this small benchmark. The 8B model is excellent on short snippets, but the known risk is long-form text where it can over-edit or hallucinate. The 70B model is still very fast and gives more safety margin for emails, prose, and larger text fields.

Benchmark Scope

The benchmark measures two paths separately:

  • Quick fix: typo correction only, scored with fragment rules for fixes, forbidden leftover typos, and preservation of voice/domain terms.
  • Smart rewrite: five-variant rewrite generation, scored for parseability, baseline typo fixes, preservation of key terms, and obvious output contract failures.

This is not a human preference eval. It is a practical regression/tuning benchmark for model choice and prompt iteration.

Quick Fix Takeaways

  • llama-3.1-8b-instant should remain a candidate for a future "ultra fast short text" mode, not the global default.
  • llama-3.3-70b-versatile is the right default when one shortcut must handle one-liners, emails, prose, and long text fields.
  • Apple Foundation should stay as a local/fallback option until prompt tuning and output cleanup reduce over-editing.

Apple Quick-Fix Tuning Follow-Up

An expanded Apple-only quick-fix corpus added mixed German/English casual text, a request-like email, a product/webhook note, and longer prose. The tuned Apple prompt improved materially on that harder set:

Apple Prompt Cases Score Accuracy Avg Latency Max Latency
Default prompt 14 101/124 81.5% 695ms 1463ms
Tuned prompt 14 118/124 95.2% 1285ms 2244ms

The remaining Apple misses were consistent: it still capitalized casual starts like hi max, translated embedded English like slowly, and reformatted shortcut listener to Shortcut-Listener. Adding more narrow instructions made the score worse, which suggests prompt-only tuning is near its useful limit.

The current tuned Apple prompt is about 5,017 characters, roughly 1.25k tokens by a simple 4-chars-per-token estimate. The shared default fast prompt is about 865 characters, roughly 216 tokens.

A no-prompt Apple baseline was also tested on the same expanded quick-fix corpus. It scored 70/124 or 56.5%, with average latency 1660ms and max latency 5795ms. Without instructions, Foundation Models frequently answered the text as an assistant, translated it, expanded todo lists into advice, or invented sign-offs. For Typofix quick fix, no-prompt Apple is not usable.

A compact Apple prompt was then tested as a speed/quality compromise. It is about 1,895 characters, roughly 473 tokens. On the expanded quick-fix corpus it scored 114/124 or 91.9%, with average latency 831ms and max latency 1787ms. This misses a strict 800ms max target, but it is close on average and much faster than the full tuned prompt. Typofix now uses the compact Apple prompt as the Apple quick-fix default, while the full tuned prompt remains available in the catalog for benchmark comparison.

Smart Rewrite Takeaways

  • For default smart rewrite, prefer Claude Haiku 4.5 or GPT-OSS 120B; both are much faster than Sonnet while keeping quality high.
  • Claude Sonnet 5 is too slow for the current interaction loop. It may still fit an explicit "highest quality" path, but not the default.
  • Claude Fable 5 was removed from the catalog after the benchmark because it was slower than Sonnet and did not justify its place in the menu.
  • Creativity is under-measured here. The next benchmark should include a small human-rated set for variation quality, not only typo/preservation rules.

Suggested Direction

  • Keep quick fix default on llama-3.3-70b-versatile for now.
  • Consider a routing rule later: short snippets can use 8B, longer text stays on 70B. Do this only after adding long-text hallucination cases to the benchmark.
  • For smart rewrite, test Claude Haiku 4.5 vs GPT-OSS 120B with human preference examples. The current score says both are plausible.
  • Continue Apple Foundation tuning for quick fix first. The likely next work is Apple-specific pre/post processing: stricter prompts, fenced-output stripping, and maybe a local diff guard that rejects outputs changing too many non-typo tokens.
  • Do not tune Apple smart rewrite until its output contract is stable. It currently fails the product semantics by translating and changing casual/domain language.
